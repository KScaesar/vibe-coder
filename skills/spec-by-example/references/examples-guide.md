## 1. Examples 表格格式

### 基本格式（三欄）

| 條件（Given） | 操作（When） | 預期結果（Then） |
|---|---|---|
| 具體的前置狀態 | 使用者執行的動作 | 系統產生的結果 |

### 進階格式（多欄條件）

| 帳戶狀態 | 帳戶餘額 | 操作 | 預期結果 | 備註 |
|---|---|---|---|---|
| 正常 | 1000 | 提款 500 | 成功，餘額 500 | Happy Path |
| 正常 | 1000 | 提款 1000 | 成功，餘額 0 | 邊界：剛好領完 |
| 正常 | 1000 | 提款 1001 | 失敗：餘額不足 | Negative |
| 正常 | 0 | 提款 1 | 失敗：餘額不足 | Edge Case |
| 凍結 | 1000 | 提款 500 | 失敗：帳戶已凍結 | Negative |

## 2. Given-When-Then 格式

### 基本結構

```gherkin
Scenario: [清楚描述這個情境]
  Given [系統的初始狀態]
  And   [額外前置條件]（可選）
  When  [使用者執行的操作]
  Then  [系統的回應/結果]
  And   [額外的驗證點]（可選）
```

### 使用 Scenario Outline（多筆資料）

```gherkin
Scenario Outline: 依不同折扣碼計算結帳金額
  Given 購物車金額為 <原始金額>
  And   輸入折扣碼 <折扣碼>
  When  使用者點擊「確認結帳」
  Then  最終應付金額應為 <最終金額>

  Examples:
    | 原始金額 | 折扣碼     | 最終金額 |
    | 1000    | SAVE10    | 900     |
    | 1000    | SAVE50PCT | 500     |
    | 500     | SAVE10    | 490     |
    | 200     | SAVE300   | 0（不能為負）|
```

## 3. 完整範例：電商結帳折扣

### 功能背景

```
功能：結帳時套用折扣碼
作為：購物用戶
我想要：在結帳時輸入折扣碼
以便於：以優惠價格完成購買
```

### 業務規則

1. 折扣碼有效期限內才能使用
2. 每筆訂單只能使用一組折扣碼
3. 折扣後金額最低為 0，不能為負
4. 已使用的一次性折扣碼不能再次使用
5. 折扣碼不適用於已特價商品（需確認）

### Examples 表格

| 折扣碼狀態 | 折扣碼類型 | 原始金額 | 操作 | 預期結果 |
|---|---|---|---|---|
| 有效 | 固定金額 -100 | 500 | 套用 | 應付 400 |
| 有效 | 九折 | 1000 | 套用 | 應付 900 |
| 有效 | 固定金額 -600 | 500 | 套用 | 應付 0（不為負）|
| 已過期 | 任意 | 500 | 套用 | 失敗：折扣碼已過期 |
| 已使用（一次性）| 固定金額 | 500 | 套用 | 失敗：折扣碼已使用 |
| 不存在 | — | 500 | 套用 | 失敗：折扣碼不存在 |
| 有效 | 固定金額 | 500 | 套用第二組碼 | 失敗：已有折扣碼 |

### Given-When-Then

```gherkin
Scenario: 成功套用有效的固定金額折扣碼
  Given 購物車中有金額 500 的商品
  And   折扣碼 "SAVE100" 為有效、未使用、折扣 100 元
  When  使用者在結帳頁輸入 "SAVE100" 並點擊套用
  Then  頁面顯示折扣 -100 元
  And   最終應付金額更新為 400 元

Scenario: 折扣後金額不可為負
  Given 購物車中有金額 200 的商品
  And   折扣碼 "MEGA500" 為有效、折扣 500 元
  When  使用者套用 "MEGA500"
  Then  最終應付金額顯示為 0
  And   系統不允許金額出現負值

Scenario: 使用已過期的折扣碼
  Given 購物車中有金額 500 的商品
  And   折扣碼 "OLD2023" 已於 2023-12-31 過期
  When  使用者套用 "OLD2023"
  Then  系統顯示錯誤訊息「折扣碼已過期」
  And   購物車金額維持 500 不變
```

### Open Questions

- [ ] 特價商品是否可以疊加折扣碼？
- [ ] 折扣碼是否有最低消費限制？（例如：滿 300 才可使用）
- [ ] 是否支援多個折扣碼同時使用？
- [ ] 退款時，折扣碼是否退還可重複使用？

## 4. 完整範例：使用者帳戶提款

### 功能背景

```
功能：ATM 提款
作為：銀行帳戶持有人
我想要：從 ATM 提取現金
以便於：在需要時取得現金
```

### 業務規則

1. 提款金額不得超過帳戶餘額
2. 帳戶凍結時不允許任何提款
3. 單日提款上限為 20,000 元
4. 提款金額必須為 1,000 的倍數

### Examples 表格

| 帳戶狀態 | 餘額 | 今日已提 | 提款金額 | 預期結果 |
|---|---|---|---|---|
| 正常 | 5000 | 0 | 3000 | 成功，餘額 2000 |
| 正常 | 5000 | 0 | 5000 | 成功，餘額 0 |
| 正常 | 5000 | 0 | 6000 | 失敗：餘額不足 |
| 正常 | 50000 | 0 | 25000 | 失敗：超過單日上限 |
| 正常 | 50000 | 15000 | 6000 | 失敗：超過單日上限 |
| 正常 | 50000 | 15000 | 5000 | 成功 |
| 正常 | 5000 | 0 | 1500 | 失敗：非 1000 的倍數 |
| 凍結 | 5000 | 0 | 1000 | 失敗：帳戶已凍結 |

### Given-When-Then

```gherkin
Scenario: 成功提款
  Given 帳戶狀態為正常
  And   帳戶餘額為 5000 元
  And   今日已提款 0 元
  When  使用者提款 3000 元
  Then  交易成功
  And   帳戶餘額更新為 2000 元
  And   ATM 吐出 3000 元

Scenario: 提款超過單日上限
  Given 帳戶狀態為正常
  And   帳戶餘額為 50000 元
  And   今日已提款 15000 元
  When  使用者提款 6000 元
  Then  交易失敗
  And   顯示訊息「已超過單日提款上限 20000 元」
  And   帳戶餘額不變
```

## 5. 完整範例：權限控管

### 功能背景

```
功能：後台文章管理權限
作為：系統管理員
我想要：依角色控制文章的操作權限
以便於：保護內容不被未授權的修改
```

### Examples 表格

| 使用者角色 | 操作 | 文章狀態 | 預期結果 |
|---|---|---|---|
| Admin | 讀取 | 任意 | 允許 |
| Admin | 編輯 | 任意 | 允許 |
| Admin | 刪除 | 任意 | 允許 |
| Editor | 讀取 | 任意 | 允許 |
| Editor | 編輯 | 自己的文章 | 允許 |
| Editor | 編輯 | 他人的文章 | 拒絕：403 |
| Editor | 刪除 | 任意 | 拒絕：403 |
| Viewer | 讀取 | 已發布 | 允許 |
| Viewer | 讀取 | 草稿 | 拒絕：404 |
| Viewer | 編輯 | 任意 | 拒絕：403 |
| 未登入 | 任意操作 | 任意 | 重導至登入頁 |

## 6. 常見 Edge Case 清單

在三角色審視時，QA 應特別關注：

### 數值邊界
- [ ] 最大值（例：餘額剛好等於提款金額）
- [ ] 最小值（例：金額為 0）
- [ ] 超出上限（例：金額比上限多 1）
- [ ] 負數輸入

### 狀態邊界
- [ ] 資源剛好被其他操作佔用
- [ ] 操作在過渡狀態時（例：剛好到期）
- [ ] 空值 / 未初始化

### 權限邊界
- [ ] 未登入
- [ ] 已登入但無權限
- [ ] 有部分權限（只能操作自己的資源）

### 時間相關
- [ ] 到期時間的臨界點（例：今天最後一秒）
- [ ] 跨日、跨月、跨年的計算

### 重複操作
- [ ] 重複提交（double submit）
- [ ] 相同操作連續執行

## 7. Open Questions 模板

每個 Spec 文件結尾應包含：

```markdown
## 待釐清問題（Open Questions）

| # | 問題 | 類型 | 負責確認 | 狀態 |
|---|---|---|---|---|
| 1 | [業務問題] | 業務規則 | PO | ⬜ 待確認 |
| 2 | [技術限制] | 技術約束 | Dev | ⬜ 待確認 |
| 3 | [測試疑慮] | 測試覆蓋 | QA | ⬜ 待確認 |
```

問題類型參考：
- **業務規則**：政策、流程、例外處理
- **技術約束**：系統限制、效能需求、整合限制
- **測試覆蓋**：難以自動化、需要手動驗證的情境
- **UI/UX**：介面行為、錯誤訊息文案
